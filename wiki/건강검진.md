# 건강검진 (Lint)

본 위키의 3가지 운영 모드 중 하나. [[CLAUDE\|운영 규칙]] 에 정의된 절차로, **위키가 시간 흐를수록 썩지 않게 유지** 하는 메커니즘. [[LLM_Wiki_개념]] 의 Lint 모드 구현.

## 점검 항목

사용자가 "건강검진 해줘" 같은 요청을 하면 Claude 가 다음을 검사:

| 항목 | 의미 | 조치 |
|------|------|------|
| **깨진 `[[위키링크]]`** | 대상 페이지 부재 | 페이지 생성 / 링크 수정 / 링크 제거 중 선택 |
| **중복/유사 페이지** | 같은 주제 두 페이지 | 병합 후보 보고 → 사용자 승인 후 병합 |
| **index 누락** | wiki 안에 있지만 [[index]] 에 없는 페이지 | 적절한 카테고리에 추가 |
| **카테고리 부적절** | 잘못 분류된 페이지 | 재분류 제안 |
| **스텁 페이지** | 본문 빈약 / 출처 없음 / 너무 오래됨 | 보강 / 삭제 결정을 사용자에게 |
| **고립 노드** | 어떤 페이지에서도 링크되지 않는 페이지 | 적절한 페이지에서 교차참조 추가 |

→ 결과는 **보고서 형태로 제시** 하고 사용자 승인 후 정리. 임의로 페이지 지우거나 병합하지 않음.

## 왜 필요한가

[[지식의_복리]] 가 성립하려면 위키가 일관성을 유지해야 함. 위키가 커지면 자연스럽게 발생하는 **드리프트** 를 잡지 않으면 결국 RAG 보다 못한 상태로 후퇴:
- 깨진 링크 → 신뢰성 하락
- 중복 → 다음 ingest 시 어디에 흡수할지 혼란
- 고립 노드 → 위키 그래프의 가치 손실
- 스텁 → "다 정리되어 있다" 는 환상 → 환각

## 의외의 도메인 동형성

이 점검 패턴은 본 위키 고유가 아니라, [[정보_일원화_패턴]] 의 모든 도메인에서 동일하게 등장:

| 도메인 | 건강검진의 형태 |
|--------|----------------|
| **위키** | Lint — 깨진 링크, 중복, 고아 페이지, 스텁 |
| **ERP** | 정합성 점검 — 전표 불일치, 마감 예외, 미반영 데이터 |
| **PMS** | 마감 점검 — 손익 일치 여부, 외주 미입력, 진행률 누락 (→ [[건설_PMS]]) |
| **MCP/AI 도구** | MCP 서버 활성화 검증, postgres 연결 살아있는지 등 |

→ 어떤 도메인이든 "정보 일원화" 시도하면 자동으로 따라오는 메커니즘. 일원화 + 건강검진 은 한 묶음.

## 본 위키 운영 권장 주기

- **주간**: 깨진 링크, 신규 추가 페이지의 index 반영
- **월간**: 중복/스텁/고립 노드 확인
- **분기**: 카테고리 구조 재검토, 출처 갱신

자료가 많이 들어온 직후에는 즉시 한 번 돌리는 게 좋음.
