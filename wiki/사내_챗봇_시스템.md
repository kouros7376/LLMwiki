# 사내 챗봇 시스템 — 300명 사무소용 개발 가이드

기존 Gemini File Search 기반 매뉴얼 챗봇(haema-knowledge-base)에서 다음 단계로 본격적인 사내 챗봇 시스템을 구축할 때 반드시 알아야 할 영역 정리. RAG·LLM 백엔드·한국어 처리·인증·운영 전 영역 + 한국 기업 사례 9건. [[Wiki_Backed_Chatbot]] 의 전제 환경이며, [[자동화_시나리오]] 의 다음 단계 인프라.

## 5계층 아키텍처 — 2025년 표준

2025년 기업용 챗봇은 **모듈형 RAG**가 baseline. 단순 LLM 래퍼는 사내 매뉴얼·도면·계약·PMS 데이터 활용 불가 → 사내 챗봇 가치 거의 없음. 성공한 엔터프라이즈 LLM 배포의 약 80%가 견고한 RAG 아키텍처 의존(Glean, TechTarget). 표준 5계층:

```
[Client UI] ←WS← [API Gateway] → [Application Layer (FastAPI)]
                                       → [LLM Gateway]
                                       → [Vector DB / Cache / Object Store]
```

각 계층 독립 확장·교체 가능해야 LLM 공급사 변경(OpenAI ↔ Claude ↔ HyperCLOVA)에도 백엔드 코드 재작성 불필요. 2025 후반부터 **MCP (Model Context Protocol)** 가 컨텍스트 일관성·툴 호출 표준 정착 → RAG 가 단순 "검색 후 생성"에서 **Context Engine** 으로 진화 중.

→ 본 사무소 현 단계 `haema-knowledge-base` 는 Gemini File Search SaaS RAG 의존이지만, 다음 MVP 는 **FastAPI 백엔드 + 자체 벡터DB + LLM Gateway 추상화**로 옮겨야 PMS·ERP·다우오피스 데이터까지 흡수 가능.

## 데이터 파이프라인 — 2025 baseline

| 항목 | 권장 값 |
|------|---------|
| 청크 크기 | 400-512 토큰 (한국어 기준 800-1000자) |
| 오버랩 | 10-20% (50-100 토큰) |
| 분할 방식 | `RecursiveCharacterTextSplitter` 시작 → Semantic Chunking 도입 검토 |
| Semantic Chunking 효과 | 단순 분할 대비 +9% Recall, 일부 도메인 +70% 정확도 (비용·복잡도 ↑) |
| 검색 방식 | **Hybrid Search** (Dense vector + BM25 + RRF) — 순수 벡터 대비 15-30% 향상 |
| 한국어 BM25 | **Kiwi** 토크나이저 (테디노트 등 다수 사례) |

**본 사무소 적용**: 매뉴얼 PDF 는 표·도면 캡션 많아 **구조 인식형 파서**(Kordoc, Deep Parser) 적합. 특히 HWP 처리는 LangChain·LlamaIndex 기본 미지원 → `Kordoc` 등 한국어 특화 OSS 거의 필수. → [[한국어_LLM_스택]] 별도 정리.

## RAG vs Fine-tuning vs Agentic — 사내 챗봇은 RAG 90%

| 방식 | 적합 도메인 | 운영 부담 | 환각 | 한국 기업 채택률 |
|------|-------------|-----------|------|-----------------|
| **RAG** | 사내 매뉴얼·정책·FAQ | 중간 | 낮음 (출처 인용) | 매우 높음 (~80%) |
| Fine-tuning | 톤·스타일 통일, 분류 | 높음 (재학습) | 데이터 의존 | 낮음 (특수) |
| Agentic | 다단계 업무·툴 호출 | 매우 높음 | 중간 | 증가 중 (~20%) |
| Hybrid | RAG + 툴 호출 | 매우 높음 | 낮음 | 신규 표준화 중 |

Skelter Labs · KB국민카드 · A언론사 사례 모두 사내 지식 챗봇은 RAG 압도적. Fine-tuning 은 비용 대비 효과 낮아 톤·말투 통일 절실 시만 보조. Agentic 은 PMS·ERP·승인 워크플로우 자동화에 유효하나 **HITL(Human-in-the-Loop) 게이트** 필수 (LLM 환각 한 번이 결재선 잘못 태움).

## LLM 백엔드 — Hybrid Routing (80/15/5)

GPU 없는 사무소가 100% 온프레미스 LLM 비현실적. 2025 한국어 챗봇 아레나 Gemini 3 시리즈 1~3위 독점. KMMLU 기준 GPT-4o(81.1) > SKT A.X 4.0(78.3) > Solar Pro 2(31B 로 Claude 4 Sonnet급)(MSAP, Elice 벤치마크).

**Microsoft Hybrid LLM (ICLR 2024) + Requesty 사례**: 라우터가 쿼리 난이도 판별 → 단순은 작은/저렴 모델, 복잡은 GPT-4o/Claude 로 라우팅 → **AI 비용 39% 절감 + 단순 쿼리 지연 32-38% 단축**.

→ 본 사무소 권장 라우팅:
- **80% 일반 질의** → Gemini 1.5 Flash ($0.075/1M, 가성비 최강)
- **15% 복합 추론·법규 해석** → Claude Sonnet ($3, 설계 도면 캡션·복합 추론)
- **5% 민감 정보 (인사·계약)** → Solar Pro 2 또는 HyperCLOVA X (국내 리전, KISA 보안 적합성)

자세한 모델 비교는 [[한국어_LLM_스택]].

## 인증·권한·감사 — 협상 불가

2025 엔터프라이즈 챗봇은 SSO + RBAC + 감사 로그 사실상 의무 (EMQX, MakinaRocks Runway, LoginRadius):

- **SSO**: SAML 2.0 또는 OIDC. 본 사무소 다우오피스(Google Cloud 구축형)이라면 **Google Workspace SSO** 직접 통합이 가장 매끄러움
- **RBAC**: 부서·민감도별 인덱스 분리 + 메타데이터 필터(예: `department:design AND security_level<=2`)
- **감사 로그**: 모든 질의·답변·검색 결과·소스 문서 ID 90일 이상 보존 (Postgres `audit_log` + Loki/Elasticsearch)
- **2026 트렌드**: 비휴먼 ID(API·봇)가 휴먼 ID 대비 3:1 증가 → **봇 자체 권한 관리** 보안 1순위

→ 챗봇이 PMS 프로젝트 정보 조회 시 "이 사용자가 이 프로젝트의 PM 또는 본부장인가" 매 질의 검증하는 권한 체크 미들웨어를 FastAPI Dependency 로 구현 필수. **RBAC 누락은 가장 흔한 사내 챗봇 사고 원인**.

## 운영·모니터링·비용 관리 — 4대 요소

1. **토큰 비용 모니터링**: RAGAS TokenUsageParser, LangSmith, Helicone, Langfuse 중 택1. 프롬프트 변경으로 평균 토큰 30% 늘면 청구서 오기 전에 CI 에서 잡혀야 함
2. **환각 측정**: RAGAS Faithfulness/Answer Relevancy 4지표 (Ground truth 없이 측정 가능). 단순 검색형은 효과적이지만 복잡 질의는 부족 → **HaluGate** (vLLM blog 2025/12) 같은 토큰 단위 환각 감지 보강
3. **사용자 피드백 루프**: 답변별 👍/👎 + 필수 reason 코멘트 → 일주일치 모아 RAGAS 재평가 + 청크/프롬프트/리랭커 튜닝
4. **A/B 테스트**: LLM 모델별·프롬프트별·청크 크기별 트래픽 분배(예: 10% 카나리)

업계 규범: **데이터 준비에 프로젝트 시간 50-70%**. MIT 연구는 **GenAI 프로젝트 95% 가 의미 있는 가치 창출 실패** (Forbes/S&P Global). 운영 지표 없이는 "쓸만한 챗봇" 인지조차 판별 불가.

## 한국 기업 사내 챗봇 사례 9건

### Skanska Sidekick (글로벌 건설사 27,000명)
- Microsoft Azure + GPT-4, 폐쇄형 클라우드
- 사내 지식 검색·문서 요약·콘텐츠 생성
- **건설업계 시사점**: 27,000 vs 300 명이지만 핵심 콘텐츠(매뉴얼·표준 도면·법규) 동일. Sidekick 5계층 아키텍처를 그대로 축소 적용 가능

### KB국민카드 (Skelter Labs 협업)
- 200건 이상 이벤트 정보 신속·정확 제공
- RAG + Extra Data 꼬리표 (시나리오별 분기)
- **교훈**: 비즈니스 목표 명확화 + 고품질 소스 콘텐츠 확보가 RAG 정확도 결정

### A언론사 (Skelter Labs)
- 검증된 출처 기반 환각 최소화. 정제 기사만 인덱싱, 출처 강제 인용
- **교훈**: 소스 콘텐츠 품질 = 챗봇 품질 (1:1)

### 삼성전자 "삼성 가우스" + "code.i" (2023.11)
- 자체 개발 텍스트·코드·이미지 생성형 AI
- 이메일 초안·문서 요약·번역 + 사내 코드 어시스턴트
- **교훈**: 자체 LLM 은 데이터 보호·도메인 적합성 우위지만 30만 명이라 가능. 300명 사무소는 자체 모델 학습 대신 클라우드 API + 강한 RAG 가 ROI 합리적

### 현대백화점 "루이스(Lewis)" — HyperCLOVA X
- 판촉 행사 카피라이팅 자동화. NAVER CLOVA Studio
- **교훈**: 한국어 카피·문체에서는 국내 LLM 이 글로벌 대비 자연스러움 우위

### 현대자동차그룹 이노션
- 사내 GPT 기반 자체 솔루션. 마케팅 전략 수립
- **교훈**: 도메인 데이터 + 일반 LLM 결합으로 충분

### LG전자 AICC + LG디스플레이 OLED
- STT/TA + 보이스봇 + 제조 데이터 분석 AI
- **성과**: 품질 이상 분석 시간 3주 → 2일
- **교훈**: 챗봇이 "대화형 UI" 머무르지 않고 운영 데이터와 결합할 때 가치 극대화

### Upstage Solar Pro 2 (조달청 1호)
- 2025.12 조달청 "생성형 AI 업무지원 서비스" 1호 공급사
- 100만 토큰당 $0.5 (동급의 1/10), 31B 컴팩트, 한국어 SOTA 근접
- **시사점**: 공공·민간 조달 환경 국내 LLM 채택 빠르게 증가 → **공공 발주 비중 높은 본 사무소도 적합**

### 카사코리아 — Amazon Bedrock 기반 Agentic
- 대고객이지만 Agentic 구조 참고

### 실패 사례 — DPD 챗봇 욕설, Taco Bell 18,000컵 주문 크래시
- 공통 원인: 가드레일 부재, HITL 부재, 도메인 외 질의 처리 미설계
- **교훈**: 챗봇은 **거절(Refuse) 능력** 핵심. "이 질문 답변할 수 없습니다" 우아하게 말하는 게 답변 정확도보다 중요. **MIT: GenAI 95% 가치 창출 실패**, **S&P Global: 2025 기업 42% 대부분 AI 이니셔티브 폐기**

## 권장 개발 로드맵 (300명 사무소)

### Phase 0 — 현재 (이미 완료)
- Gemini File Search 기반 `haema-knowledge-base` 운영
- 사내 매뉴얼 PDF Q&A
- 검증된 가치: 직원이 "AI 에게 묻는다" 행동 정착

### Phase 1 — PoC (4-8주)
- FastAPI + Chainlit 베이스라인
- Synology Docker 에 **Qdrant** 또는 **Postgres+pgvector** 구동
- PDF + HWP 처리 파이프라인 (`Kordoc` 또는 `Deep Parser`)
- 임베딩 = OpenAI `text-embedding-3-small` (저비용 시작) 또는 `KURE-v1` 자체 호스팅
- LLM = Gemini 1.5 Flash (가성비)
- Hybrid Search (Kiwi BM25 + dense)
- 인력: IT 책임자 1 + Claude Code 보조
- 산출: 단일 본부 30명 베타, RAGAS 점수 baseline
- 예산: API 월 $50-200 + Synology 디스크 추가

### Phase 2 — MVP (8-16주)
- Google Workspace SSO 통합
- RBAC: 부서·민감도 메타데이터 필터
- 감사 로그: Postgres `audit_log` + 90일 보존
- LLM Gateway 추상화 (Hybrid LLM Routing)
- 사용자 피드백 (👍/👎 + reason)
- 증분 인덱싱 자동화 (Synology 폴더 변경 감지 → Qdrant 업데이트)
- 모니터링: Langfuse 또는 Helicone
- 인력: IT 책임자 + 외주 LLM 엔지니어 1 (3개월)
- 산출: 5개 본부 100명 베타, RAGAS Faithfulness ≥ 0.85
- 예산: API 월 $300-800 + 외주 ₩30-50M

### Phase 3 — 정식 (3-6개월 추가)
- PMS·ERP·다우오피스 API 통합 (Tool 계층)
- HITL 게이트 있는 Agent (예: 외근 신청서 초안·결재 라우팅)
- A/B 테스트
- SLA 99.5% (Synology HA + 클라우드 LLM 폴백)
- 분기별 RAGAS 재평가
- 운영 0.5 FTE 정착

## 핵심 의사결정 체크리스트

1. **현 Gemini File Search → 자체 RAG 전환 시점**: 사용자 100명 도달 또는 PMS/ERP 데이터 결합 욕구 발생 시
2. **벡터DB**: Synology만 → **Qdrant Docker**, RDB 통합 필요 → **pgvector**
3. **임베딩**: OpenAI `text-embedding-3-small` 시작 → 안정화 후 **KURE-v1** 자체 호스팅
4. **LLM**: 80/15/5 라우팅 (Gemini Flash / Claude Sonnet / Solar·HyperCLOVA)
5. **HWP**: `Kordoc` 또는 `Deep Parser` — 한국어 RAG 숨은 결정 변수
6. **UI**: Chainlit (인증·스레드·피드백 내장)
7. **인증**: Google Workspace SSO (다우오피스와 동일 IdP)
8. **감사**: Postgres `audit_log` + 분기별 검토
9. **평가**: RAGAS + 사용자 피드백 + 월별 회고
10. **거버넌스**: 가드레일 프롬프트 + Refuse 시나리오 + HITL 게이트
