# 자동화 시나리오 — 건축설계사무소 IT 담당자용 5개 + 로드맵

[[워크플로우_자동화_도구]] 를 실제 사내 업무에 적용하는 구체적 시나리오. 우선순위와 단계적 도입 로드맵 포함.

## 도구 분기 — RPA · 워크플로우 자동화 · LLM 에이전트

본 페이지의 시나리오들은 다음 4가지 도구 중 적합한 것을 선택해 구현. 본 사무소 [[RPA|RPA 도입 타당성 분석]] 결과:

| 업무 성격 | 도구 |
|----------|------|
| API 있고 데이터 흐름 핵심 (다우오피스↔NAS↔PMS/ERP) | **n8n / Power Automate** ([[워크플로우_자동화_도구]]) |
| GUI만 있고 사람 클릭 필수 (정부 사이트 — 세움터·홈택스·4대보험 EDI) | **[[RPA]]** (제한적·필수 영역만) |
| 비정형 의사결정·문서 분석 | **LLM 에이전트** ([[Wiki_Backed_Chatbot]]) |
| BIM/CAD 도서 처리 | Revit API + Dynamo + Python |

→ 시나리오 1 (인허가)는 RPA + IDP, 시나리오 2/3 (보고서/온보딩) 는 워크플로우 자동화, 시나리오 5/6 (회의록·위키)는 LLM 에이전트.

## 5가지 핵심 시나리오

### 시나리오 1 — 인허가 서류 처리 자동화
**문제**: 인허가 첨부파일을 수동 정리, 반복 입력
**조합**: n8n + Power Automate (Desktop Flow) + Claude API
**흐름**:
```
메일 첨부 PDF
  → AI Builder/Claude API 데이터 추출
  → Excel/DB 정리
  → 결재 요청 자동 발송
```
**효과**: 인허가 한 달 → 보름 단축 (BIM 자동 법규체크와 결합 시) — [[정림건축_혁신본부|정림 BIM Package]] 의 BIM Law 데이터 자동 체크와 결합 가능

### 시나리오 2 — 프로젝트 진행 일일 보고서
**문제**: 매일 PMS 조회 + Excel 정리 + 메일 발송 반복
**조합**: n8n Schedule Trigger + 사내 PMS DB + ChatGPT/Claude
**흐름**:
```
오전 8시 자동 실행
  → 사내 PMS DB 조회 (postgres)
  → ChatGPT 요약
  → Teams/Slack 일일 리포트 전송
```
**효과**: [[건설_PMS]] 의 일원화된 데이터 활용 → [[정보_일원화_패턴]] 의 "한 번 쌓은 데이터로 매일 자동 의사결정"

### 시나리오 3 — 직원 온보딩
**문제**: 신입 사원 1명당 IT 셋업 5~10시간 수작업
**조합**: Power Automate + Active Directory + 365
**흐름**:
```
HR 입사 등록
  → AD 계정 생성
  → 365 라이선스 할당
  → 그룹 추가 (부서/프로젝트)
  → 환영 메일 + 첫 주 일정 자동 등록
  → 사내 위키 가이드 링크 전달
```
**효과**: 한 명당 30분 미만, 누락 0%

### 시나리오 4 — 외주 협력사 데이터 수집
**문제**: 외주 협력사(구조·설비·전기·조경·인테리어)가 사내 PMS 입력 안 함 → 데이터 빔
**조합**: n8n + Google Forms / Tally + 사내 DB
**흐름**:
```
주 1회 외주사에 폼 자동 발송
  → 응답 수집 → 사내 DB 적재
  → 누락 시 개별 알림 (관리자 + 협력사)
```
**효과**: [[건설_PMS]] 의 "외주 미입력 → PMS 형해화" 함정 완화 — 협력사 포털을 별도 시스템 만들지 않고도 효과

### 시나리오 6 — 사내 LLM 위키 + 챗봇 통합 (Wiki-Backed Chatbot)
**문제**: 기존 [[사내_챗봇_시스템|haema-knowledge-base]](Gemini RAG)는 매뉴얼 PDF 만 처리. 통찰·분석·전략 자료(현 kouwiki 26+ 페이지)는 LLM 컨텍스트로 활용 못 함
**조합**: kouwiki vault + FastAPI + Claude Sonnet API + (fallback) Gemini File Search + Caddy + LDAP
**흐름**:
```
사용자 질문
  → 위키 매칭 (Kiwi BM25 + 임베딩, 매칭 점수 임계치 평가)
     ├─ 충분 → 위키 페이지 본문 + [[교차참조]] 1-hop 컨텍스트 → Claude 답변
     └─ 미스 → Gemini File Search (사내 매뉴얼 RAG) fallback
  → 답변 + 출처 ([[페이지]] 또는 [PDF, p.12])
  → "이 답을 위키에 정착시킬까요?" 액션 (Karpathy 패턴)
  → 다우오피스 메신저 알림
```
**효과**: 통찰·분석 질의 응답 자동화 + [[지식의_복리|복리]] 누적 + [[Wiki_Backed_Chatbot|환각 -55% / 토큰 -99%]] 정량 효과 (WikiChat·Graphify 벤치마크)
**도입 단계**: [[Wiki_Backed_Chatbot|5단계 도입 로드맵]] 의 Phase 1-2 (1-2주 + 2-4주). 자세한 코드 골격 동일 페이지 참조.
**전제 인프라**: [[한국어_LLM_스택]] (Kiwi · KURE-v1 · 80/15/5 LLM 라우팅) + [[Graphify]] (Phase 3) + [[LLM_Wiki_운영_패턴|5~10 admin 큐레이션 모델]]

### 시나리오 5 — 회의록 → 위키 자동 ingest
**문제**: 회의록이 Slack/메일에 흩어져 다음 회의 때 매번 검색
**조합**: 녹음 → Whisper → Claude API → n8n → 위키
**흐름**:
```
회의 녹음 파일 업로드
  → Whisper 전사
  → Claude API 요약 (의사결정/액션아이템 분리)
  → n8n
  → kouwiki repo 자동 push (또는 Notion DB)
```
**효과**: [[지식의_복리]] 의 조직 자동 적용 — 매 회의가 위키에 누적, 다음 회의는 이전 결정 위에서 시작. **회사가 RAG → Wiki 로 전환**.

→ 본 [[CLAUDE|kouwiki 운영규칙]] 자체가 이 시나리오의 수동 버전. 자동화 시 [[더존_아마란스10|아마란스10 ONE AI 회의록 자동작성]] 의 연동형 대안.

| 시나리오 2 (일일 보고서) | 즉시 체감 | 낮음 (조회+요약) | 낮음 | ★★★ **PoC 1순위** |
| 시나리오 3 (온보딩) | 누적 큼 | 중간 (AD 권한) | 중간 (계정 실수) | ★★ |

| 시나리오 4 (외주사) | 큼 | 중간 (외부 협조) | 중간 (외주 거부감) | ★ |

## 단계적 도입 로드맵

```
1단계 (1~2개월) — PoC
└─ n8n 셀프호스트 1대 구축
└─ 시나리오 2 (일일 보고서) 1개 운영
└─ 효과 측정·문서화

2단계 (3~6개월) — 확산
└─ 효과 검증된 워크플로우 5~10개 운영
└─ 사용자 교육 + 운영 가이드 작성
└─ Power Automate 병행 검토 (RPA 영역 식별)

3단계 (6~12개월) — 표준화
└─ 사내 자동화 플랫폼 표준화
└─ 신규 요구사항 → 자동 검토 절차
└─ 워크플로우 카탈로그·문서·온보딩 정비
```

## 본 위키 컨텍스트 — 자동화의 메타 가치

### [[정보_일원화_패턴]] 의 자동화 적용
워크플로우 자동화의 본질은 "여러 시스템에 흩어진 데이터를 한 흐름으로 묶는 것" — 즉 [[정보_일원화_패턴]] 의 자동화 도메인 구현.

### [[지식의_복리]] 의 시간 차원 적용
시나리오 5(회의록 ingest)는 "조직 지식의 복리 운영"의 자동화. 매 회의가 위키에 자동 누적되면, 1년 뒤 의사결정의 지식 기반이 압도적으로 두꺼워짐.

### [[건강검진]] 의 자동화 도메인
워크플로우도 [[건강검진]] 필요:
- 실패한 실행 (실패율 5% 이상이면 알림)
- 미실행 워크플로 (예: 매일 동작해야 하는데 어제 한 번도 안 동작)
- 중복 워크플로 (같은 트리거 두 번)
- 사용 안 하는 노후 워크플로 정리

### [[건축IT_사내개발_전략]] 1단계 권장
사내 개발팀이 없는 사무소도 워크플로우 자동화는 즉시 시작 가능. 정식 사내 개발 인력 채용 전 단계의 가성비 높은 시작점.
