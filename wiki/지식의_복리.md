# 지식의 복리

> "위키는 복리로 쌓이는 자산이다. RAG처럼 매번 재발견하는 게 아니라, 한 번 이해한 것 위에 다음 이해가 쌓인다."  — Andrej Karpathy

LLM 활용에서 **RAG vs Wiki** 의 본질적 차이를 가리키는 메타 명제. 본 위키 전체의 운영 철학이며, ERP/PMS 도메인에서도 동일 패턴이 반복됨.

## 핵심 대비

| 모델 | 비유 | 한계 / 가치 |
|------|------|------------|
| **RAG** | 책을 매번 처음부터 다시 펼치기 | 지식이 쌓이지 않음. 어제 찾은 것도 오늘 또 찾음 |
| **위키 (Compiled Knowledge)** | 책에 밑줄 친 것 위에 새 메모를 추가 | 운영 비용(ingest, lint) 발생. 그러나 시간이 갈수록 가치가 복리로 증가 |

## 왜 "복리"인가

- 새 자료 ingest 시 **기존 페이지의 맥락 위에서** 흡수됨 → 단순 추가가 아닌 통합
- 자료 N개가 들어오면 정보량은 N개인데, **연결(교차참조) 수는 N²/2 까지** 커질 수 있음 → 그래프적 가치
- 본 위키 한 페이지 안에서도 다른 도메인 패턴을 끌어와 새 통찰을 만들 수 있음 → "복리"의 정량적 근거

## 본 위키에서의 구현

본 kouwiki 자체가 이 명제의 라이브 구현:
- [[CLAUDE]] 의 운영 규칙 = 복리 누적의 사전 합의
- [[LLM_Wiki_개념]] 의 Ingest/Query/Lint = 복리 운영의 메커니즘
- [[건강검진]] 의 깨진 링크/고아 페이지/스텁 점검 = 복리가 무너지지 않게 유지하는 장치

## 의외의 도메인 연결 — ERP

[[건설_PMS]] 의 **대림산업 손익 일원화 사례** 는 회계 ERP 도메인에서 같은 패턴이 발현된 것:
- "회계상 손익"과 "공사 손익"이 따로 노는 것 = 조직의 RAG 상태 (같은 사실을 두 번 해석)
- 손익 일원화 = 조직의 위키 전환 (한 번 정리한 위에 누적)
- 효과: 단위 사업장별 손익 가시화, 자료 일관성, 의사결정 신속

→ ERP 마이그레이션은 단순 "시스템 교체"가 아니라 **"조직의 RAG → 위키 전환"** 으로 재정의 가능. 자세한 매핑은 [[정보_일원화_패턴]].

## 의외의 도메인 연결 — Jira / 협업 도구

블로그 [[LLM_Wiki_개념|사례 2]] 의 작가가 회사 적용 아이디어로 든 것:
> "Jira issue 다 때려박고 ingest 한 뒤에 신규 issue 발생 시 claude에게 던질 것 같다"

이는 **레거시 협업 데이터의 위키 전환** — ERP 마이그레이션의 데이터 이관과 같은 패턴. 즉 Jira issue 이관도 [[정보_일원화_패턴]] 의 한 사례.

## 알아야 할 비용
- **Ingest 토큰 비용** — 자료 정리 시 LLM 토큰 소비 큼. 비용 모니터링 필요
- **Lint 부담** — 주기적 [[건강검진]] 안 하면 환각·드리프트 발생
- **저작권/출처 관리** — 원본 자료 보존 + 위키 페이지 출처 명시 필수

## 메타 비유 — 컴파일된 지식 ↔ 사서

본 명제는 두 가지 비유로 다른 관점에서 표현됨:

| 비유 | 관점 | 출처 |
|------|------|------|
| **컴파일된 지식 (Compiled Knowledge)** | 사용자 관점 — 매번 재해석하지 않게 미리 정리 | Karpathy LLM Wiki gist |
| **사서 (Librarian)** | 시스템 관점 — 묻기 전에 스스로 정리·연결·패턴 발견 | [[LLM_Wiki_웹구현_사례\|Abraham Song]] (Brunch, 2026) |

> "대부분의 AI 기반 지식 도구는 LLM 을 검색 엔진처럼 다룬다. 이 시스템은 LLM 을 사서처럼 다룬다. 사서는 당신이 물어볼 때까지 기다리지 않는다 — 스스로 정리하고, 연결하고, 당신이 놓친 패턴을 알아챈다." — Abraham Song

→ 컴파일이 **결과물**의 비유라면, 사서는 **행위자**의 비유. [[Graphify]] 의 자동 클러스터링·INFERRED 엣지가 사서의 능동적 정리 활동의 정량 구현 ([[LLM_Wiki_웹구현_사례]] 의 OAuth2 ↔ PaymentGateway 자동 발견 일화가 실증).

## 정량 증거 — WikiChat + Graphify

본 명제가 단순 슬로건이 아니라 정량 측정 가능한 효과임이 두 사례로 검증됨:

**WikiChat (Stanford OVAL, EMNLP 2023, Wikimedia Research Award 2024)**: Wikipedia 그라운딩 7단계 파이프라인.
- 시뮬레이션 대화 **97.3% factual accuracy**
- 인간 사용자 최근 토픽 대화 **97.9%** — **GPT-4 대비 환각 55% 감소**
- 7B LLaMA 로 distill 해도 품질 유지 → **위키 그라운딩이 환각을 정량적으로 잡는다는 학술적 증거**

**[[Graphify]] 토큰 절감**:
- 52파일 코퍼스 평균 질의당 ~1.7K 토큰(그래프) vs ~123K 토큰(원본) → **71.5×** 절감
- 더 큰 셋업(456 Obsidian 노트 + 65+ 영구 노트)에서 **499×** 절감
- → 동일 질의 100회 반복 시 LLM 비용 폭락. 본 위키가 수백 페이지 되면 즉시 환산 가능한 효과

→ 본 명제는 더 이상 "그럴듯한 비유"가 아니라 **환각 -55% / 토큰 -99% 의 측정 가능한 운영 효과**. [[Wiki_Backed_Chatbot|위키 기반 챗봇]] 도입의 가장 강한 ROI 근거.

## 한 줄 요약

**RAG = 매 회의마다 회의록 처음부터 검색하기. Wiki = 회의록 위에 결정사항을 누적해 두고, 그 결정 위에서 다음 회의를 시작하기.**
