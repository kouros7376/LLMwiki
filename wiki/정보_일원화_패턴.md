# 정보 일원화 패턴

같은 사실을 두 번 입력·해석하지 않고, **단일 기억소(single source of truth)에 한 번 정리해서 그 위에 누적** 한다는 메타 패턴. 본 위키에서 ERP·PMS·LLM 도메인을 가로지르는 공통 구조 — kouwiki 의 가장 중요한 통찰 중 하나.

## 동일 패턴의 도메인 매핑

| 도메인 | "두 번 정리하던" 비효율 | "일원화" 해법 | 본 위키 페이지 |
|--------|-------------------------|--------------|----------------|
| **회계 ↔ 프로젝트** | 회계 손익 vs 공사 손익이 따로 → 준공 시 예정원가 재계산 | 대림산업 PMS 2단계 — 손익관리 일원화 | [[건설_PMS]] |
| **ERP ↔ 그룹웨어** | 결재·메일·메신저·회계가 따로 → 같은 정보 중복 입력, SSO 복잡도 | 단일 플랫폼 통합 | [[더존_아마란스10]] |
| **LLM 컨텍스트** | RAG: 매 질의마다 재검색 → 같은 자료 매번 재해석 | 한 번 ingest 후 누적 | [[지식의_복리]], [[LLM_Wiki_개념]] |
| **개인 지식** | 노션·메모·드라이브 분산 → 같은 인사이트 매번 재발견 | 단일 운영 규칙 하의 위키 | [[CLAUDE]] (본 위키) |
| **레거시 협업 데이터** | Jira/Slack/메일에 흩어진 의사결정 → 신규 issue 시 다시 추적 | 전체 ingest 후 LLM 질의 | [[LLM_Wiki_개념]] 사례 2 |

## 왜 같은 패턴인가

세 가지 공통 구조:
1. **읽기 → 이해 → 저장** 흐름을 한 번에 묶어 둠
2. 다음 작업이 처음부터가 아니라 **이전 정리 위에서** 시작
3. 실패 양상도 동일 → 모든 도메인이 [[건강검진]] 류 메커니즘을 갖춤

| 도메인 | "건강검진" 메커니즘 |
|--------|--------------------|
| 위키 | Lint — 깨진 링크, 고아 페이지, 스텁 |
| ERP | 정합성 점검 — 전표 불일치, 마감 예외 |
| PMS | 마감 점검 — 손익 일치 여부, 외주 미입력, 진행률 누락 |

## 도입 시 공통 함정

세 도메인에서 동일한 함정이 반복:

### 함정 1 — 운영 규칙 합의 부재
- 위키: Schema 없음 → 단지 "더 큰 RAG"가 됨
- ERP: 업무 규칙 합의 없이 도입 → 부서별 우회 운용
- PMS: 외주 협력사 입력 규칙 없음 → 데이터 빔

### 함정 2 — 입력 부담의 사용자 전가
- 위키: ingest 게으름 → 자료 안 들어옴
- ERP: 직원 메신저 회피 → 그룹웨어 형해화
- PMS: 외주사 미입력 → 일원화 무너짐

### 함정 3 — 옛 시스템 종료 실패
- 위키: 기존 메모/드라이브 못 끄면 이중 운영
- ERP: 마이그레이션 후 옛 ERP 못 끄면 이중 운영 (가장 흔한 ERP 실패 사유)
- PMS: 엑셀 관행 그대로 → PMS는 "보고용 화면"으로 전락

→ 세 함정 모두 [[ERP_RFP_체크리스트]] 의 운영·문화 항목으로 반영.

## 시사점 (전산실 총괄 관점)

**ERP 마이그레이션을 단순 "시스템 교체" 가 아닌 "조직의 RAG → 위키 전환" 으로 재정의** 하면 의사결정의 우선순위가 달라진다:

| 일반적 우선순위 | 일원화 패턴 우선순위 |
|----------------|---------------------|
| UI 화려함 / 기능 가짓수 | **규칙 합의 + 일원화 가능성** |
| 모듈 풍부함 | 옛 시스템 종료 가능성 |
| 가격 | 입력 부담 분산 가능성 |
| AI 기능 | 건강검진 자동화 |

이 관점에서는 [[더존_아마란스10]] 의 "단일 로그인 통합" 가치가 단순 편의가 아니라 **일원화 패턴의 한 구현**. 반대로 [[영림원_K-System_Ace]] + 외부 도구 조합도 SSO·정합성 자동화로 **일원화 가능성** 을 확보하면 동등한 가치 제공 → [[통합형_vs_연동형]].

## kouwiki 자체에 적용

본 위키 자체가 이 패턴의 라이브 구현이며 동시에 사용자 본인의 ERP 마이그레이션 의사결정을 위한 메타 도구. 즉:
- **수단**: 본 위키 = 본인 지식의 일원화
- **목적**: 회사 ERP 마이그레이션 = 회사 정보의 일원화
- 두 작업의 본질이 같으므로, 본 위키 운영에서 배우는 교훈(예: ingest 부담, lint 자동화)이 ERP 의사결정에 직접 적용됨
