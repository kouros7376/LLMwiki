# kouwiki — 개인 지식 위키 운영 규칙 (Schema)

이 저장소는 **kouwiki** 라는 개인 지식 위키이다.
Karpathy의 LLM Wiki 아이디어(<https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f>)를 참고해 구현했다.

운영 모델: 사용자가 `raw/`에 자료를 넣고, Claude가 그것을 읽어 `wiki/`에 정리·요약·교차참조한다. 사용자는 필요할 때 위키에 질문을 던진다.

이 파일은 **운영 규칙(Schema)**이다. Claude는 이 위키를 다룰 때 항상 이 규칙을 따른다.

## 사용자 컨텍스트

- 역할: 약 300명 규모 건축설계사무소의 **전산실 총괄**
- 주된 관심 주제 (초기 카테고리)
  - **AI 바이브 코딩** — 동향 기사, 신규 도구 업데이트
  - **ERP / 마이그레이션** — 통합 ERP 마이그레이션 업계 정보 (더존 아마란스, 영림원 등)
  - **전산업무 자동화** — 업무 자동화 도구 레퍼런스
  - **Claude 활용** — Claude 신규 버전 / 사용법

새로운 주제가 들어오면 사용자와 논의 후 카테고리를 추가한다.

## 폴더 구조

```
kouwiki/
├── CLAUDE.md          ← 본 파일. 운영 규칙
├── raw/               ← 원본 자료. 사용자만 추가/수정. Claude는 읽기 전용
└── wiki/              ← 정리된 지식. Claude가 작성/관리
    ├── index.md       ← 전체 목차 (카테고리별)
    ├── log.md         ← 작업 이력
    └── *.md           ← 주제별 위키 페이지 (플랫)
```

- `raw/` 와 `wiki/` 모두 **하위 폴더 없이 플랫**하게 관리한다.
- 분류는 폴더가 아니라 `wiki/index.md`의 카테고리 섹션으로 한다.

## 운영 방법 (3가지)

### 1. 자료 넣기 (Ingest)

사용자가 `raw/`에 파일을 넣고 "자료 넣었어" 또는 파일명을 알려주면 Claude는:

1. `raw/<파일명>`을 읽는다.
2. 핵심 내용을 추출해서 적절한 위키 페이지에 반영한다.
   - 신규 주제 → `wiki/<주제명>.md` 신규 생성
   - 기존 주제 보강 → 해당 페이지에 섹션 추가/갱신
3. **교차참조를 적극적으로 활용한다.** 본문 안에서 다른 주제와 겹치는 단어가 보이면 즉시 `[[위키링크]]`로 연결한다. 신규 페이지는 최소 1개 이상 기존 페이지로 연결되어 있어야 한다 (고립 노드 금지).
4. `wiki/index.md` 의 해당 카테고리에 페이지를 추가한다.
5. `wiki/log.md` 에 한 줄 이력을 남긴다 (날짜 / raw 파일 / 갱신된 페이지).

자료가 URL이면 사용자가 본문을 raw/에 붙여넣거나, Claude가 WebFetch로 가져온 뒤 raw/에 저장한다.

### 2. 질문하기 (Ask)

사용자가 위키에 질문하면 Claude는:

1. `wiki/index.md` 로 관련 페이지를 식별한다.
2. 해당 페이지 + 그 페이지가 링크하는 `[[교차참조]]` 페이지들을 읽는다.
3. 답변에 근거 페이지를 `[[위키링크]]` 로 인용한다.
4. 위키에 정보가 없으면 **"위키에 없음"** 이라고 명확히 말하고, 필요 시 `raw/` 까지 뒤져 본다.

### 3. 건강검진 (Health Check)

사용자가 "건강검진 해줘" 같은 요청을 하면 Claude는:

- 깨진 `[[위키링크]]` (대상 페이지 부재) 탐지
- 중복/유사 페이지 식별 (병합 후보)
- `index.md` 에서 누락되거나 카테고리가 부적절한 페이지
- 너무 오래되었거나 출처 불명, 본문이 빈약한 스텁 페이지
- 고립 노드 (어떤 페이지에서도 링크되지 않는 페이지)

→ 결과를 보고서 형태로 제시하고, **사용자 승인 후** 정리한다. 임의로 페이지를 지우거나 병합하지 않는다.

## 페이지 작성 규칙

모든 위키 페이지는 다음 구조를 따른다:

```markdown
# <페이지 제목>

<요약 1~3문장 — 이 페이지가 무엇을 다루는지>

## <본문 섹션>
...

## 출처
- raw/<파일명>  또는  URL
- (수집일: YYYY-MM-DD)

## 관련
- [[페이지A]] — 한 줄 설명
- [[페이지B]] — 한 줄 설명
```

- 파일명은 페이지 제목과 일치 (공백은 `_` 로 대체)
- 본문 섹션은 H2 (`##`) 단위로 나눈다
- 본문 안에서도 다른 주제가 등장하면 `[[위키링크]]` 로 만든다 — `## 관련` 섹션에만 모아두지 말 것
- 코드/명령어/설정값은 코드블록으로 표기

## [[위키링크]] 표기 규칙

- `[[페이지명]]` 은 vault 내 `페이지명.md` 를 가리킨다 (Obsidian 기본 동작).
- 표시 텍스트를 다르게 쓰려면 `[[페이지명|보여줄 텍스트]]`.
- 파일명은 한글 허용. 공백은 `_` 로 대체.

## Claude의 행동 원칙

- 사용자 명시 요청 없이 `raw/` 안의 파일을 수정/삭제하지 않는다.
- `wiki/` 페이지를 지우거나 병합할 때는 사용자 확인을 받는다.
- 자료를 정리할 때 **요약**이지 단순 복붙이 아니다. 핵심만 추리되, 출처는 항상 명시한다.
- 사용자의 역할(전산실 총괄, 건축설계사무소)을 고려해, 실무에서 쓸 수 있는 형태로 정리한다 — 일반론보다 적용 포인트.
- 모르는 것은 모른다고 한다. 위키에 없는 내용을 있는 것처럼 답하지 않는다.
